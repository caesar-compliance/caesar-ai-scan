# T005 Architectural Decisions

## 1. Zero External Dependencies & CDNs
- **Decision**: Avoid using external fonts (e.g., Google Fonts), tracking libraries (e.g., Google Analytics, Mixpanel), or CSS/JS frameworks loaded via CDN (e.g., Tailwind CSS, React, jQuery).
- **Rationale**: Upholds the strict offline, air-gapped compliance architecture of Caesar AI Scan. By avoiding third-party assets, we prevent data leakage and ensure 100% security boundaries. All dashboard renderings are handled by locally-hosted stylesheets and native JS fetch calls.

## 2. Dynamic client-side loading of Static JSON
- **Decision**: Instead of static HTML generation of tables, build the dashboard as a single page application (SPA) shell utilizing native Javascript `fetch` to read sample data files in `site/data/` dynamically.
- **Rationale**: This closely mirrors how the future production Governance OS integration works. The UI layout remains completely separated from the scan metrics data structure, allowing the dashboard code to remain clean and scalable.

## 3. GitHub Pages Deployment via GitHub Actions Workflow
- **Decision**: Use GitHub Actions to build and deploy Pages instead of committing build products to a `gh-pages` branch.
- **Rationale**: Clean separation of source files and deployment outputs. Storing raw artifacts or autogenerated `site-build.json` in the git log is avoided. The site is built dynamically in the CI runner, ensuring it's always verified by the latest program validation suite before deployment.

## 4. HTTPS Certificate Delay Acceptability
- **Decision**: Delay HTTPS enforcement validation until Let's Encrypt / GitHub DNS checks are completed, treating initial HTTPS failures due to certificate processing as external platform delays, not integration errors.
- **Rationale**: Automated SSL provisioning takes up to several hours once DNS record checks complete in the GitHub UI. Removing the CNAME or manually overriding DNS parameters during certificate provisioning degrades verification.
